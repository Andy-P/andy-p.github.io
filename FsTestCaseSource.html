Using TestCaseSoureAttribute with FsUnit in Xamarin

Recently, I attended my first F# meetup here in Japan. Actually, they don't call it a meetup. The term in Japanese is F#談話室 which roughly translates to F# conversation room. The atmosphere was pretty laid back. Many members seemed to know each other. The group was small compared to some of the other cities’ F# meetups, But it gave the gathering an intimate feeling, The attendees included F# MVPs, the translator of a number of F# programming books, and the author of the open source FSRandom library. In case you didn't know, this is also the group that came up with the idea for the F# Advent Calendar.

Since this was my first meeting, I decided to give a 'spur of the moment' lightning talk on my experience using the <TestCaseSoureAttribute> with FsUnit. The talk focused on the advantages of using it for TDD.

Go big or go home... a TDD motivating “Incident”

During my talk, I shared a story from a few years back. Basically, I completely screwed up There was an "incident" at my work. I'd been with my current employer for a little over a year and had grown "comfortable". I'd also pulled an all-nighter to finish off a tool we called getFlat.

As the name suggests, the tool was designed to look across all our positions, and then figure out the best way to get us to a zero delta (aka: having no position). Unfortunately, an error in my initialization code caused me to send in a very large number of sell orders and within two or three seconds we were very, very short the market. In the end, we were able to get us out with a minor loss that was equivalent to a single day’s average pnl.

TDD |> Like sunscreen for coding

From the pnl point of view, my mistake had been a non-event. I’d been given a gentle tap on the shoulder by the gods of trading - “... be more careful.. a lot more careful!” Soon after that I found my way to TDD. And it quickly moved into the top of my list “things to do to avoid ‘incidents’”. I have been using TDD for a while now, but until recently it felt like putting on sunscreen. You do it because you know it will protect you, but it’s not really something you enjoy.

When I began with functional programming, I tried to follow Scott Wlaschin’s advice to '...to be fluent and comfortable with functional programming in general, it is critical that you think functionally, not imperatively.' Kit Eason echos this with 'Forget everything you know about OO to start with.' Once I began following this advice, TDD became much smoother. Functional programming seems to naturally suit unit testing and TDD. One input to one output.

Nunit's TestCaseSoureAttribute with FSUnit

Recently, I went through Mark Seemann’s Outside-In TDD. The course uses C# to explore Outside-in TDD, or the London School of TDD as it is sometimes known. The core idea is that you build your tests and code from the outside-in. That is, that you begin by making an empty shell referred to as a skeleton. This skeleton is completely bare at the start. Piece by piece you fill in the necessary structures until you finally arrive at the innermost functionality. Frequently, this approach will result in less code and shorter development times because only absolutely necessary code in written at each step.

One characteristic of this approach when using C# is that you end up doing a fair amount of mocking. However, when doing this in F#, I found I barely need to use any mocking. This is partly due to the type of code I am writing. Much if my work uses functional transformations to process and analyzes data. But it also a resulted of using a functional approach.

Below, I detail one tool that can be used with this approach - TestCaseSoureAttribute. First, a bit on my environment. I'm working in Xamarin targeting Mono/.Net4.5. Xamarin includes a modified version of the NUnitLite framework and a familiar set of APIs for writing unit tests. I’m also make use of FsUnit.
