Using TestCaseSource with FsUnit in Xamarin

Recently, I attended my first F# meetup here in Japan. Actually, they don't call it a meetup. The term in Japanese is F#談話室 which roughly translates to F# conversation room. The atmosphere was pretty laid back. Many members seemed to know each other. The group was small compared to some of the other cities’ F# meetups, but it gave the gathering an intimate feeling, The attendees included F# MVPs, the translator of a number of F# programming books, and the author of the open source FSRandom library. In case you didn't know, this is also the group that came up with the idea for the F# Advent Calendar.

Since this was my first meeting, I decided to give a 'spur of the moment' lightning talk on my experience using the <TestCaseSource> attribute with FsUnit. The talk focused on the advantages of using it for certain cases of TDD.

Specifically, I found this approach useful where you have the following scenario

  • You have a state
  • This state is modified by a some stream of data
  • The modification produce some effect that you want to captured

In functional languages we can easily model this by

  • Pass in some state and modifier to a function
  • Return the modified state and accumulate a list of modifications

I find myself using this approach when I want to pipe through a sequence events by recursively iterating over a list of data events, while accumulating a list of responses. Below I show how we can use the <TestCaseSource> to create unit tests for sequences of events.

module SimpleExample = 

    let modifyStateAndProduceEffect state modifier  =  
        let newState = state * (1. + modifier)
        newState,  newState - state

    // a list of events or data points
    let ms = [-0.03; -0.02; 0.03; 0.021] // down -3%, -2%, then back up +3%, +2.1%  
     
    let rs = ms |> List.scan (fun (state, _) modifier -> modifyStateAndProduceEffect state modifier) (1., 0.)

The List.scan operation builds a list of results by iterating over a list data points, applying a function, and accumulating the results of that functions. This nicely simulates the piping a series of events through a function to get a series of results..

The results can easily be parsed using simple list operations.

Here's as example of using the TestCaseSource attribute to create unit tests for the above. We begin by defining a TestFixture and a helper function approxMatch. This function compares two lists of values to see if they are an approximate match. FsUnit already has a great set of built-in comparision functions, but for complex list tests, or tuple comparisions you may need to write your own.

[<TestFixture>]
type ``A Sample Test`` () = 

    let approxEqual ls1 ls2 e =
       List.forall2 (fun (x1:float) (x2:float) -> Math.Abs(x1 - x2) <= e) ls1 ls2

There are a few important things to note about the sample data.

  1. NUnit expects it in the form of a static member
  2. It is defined as an array of arrays i.e. [ ][ ]
  3. You can put almost any object inside the inner nested array.
  4. The contents of each of the cells inthe inner array will be passed one by one to your unit test

In this case, I uses a two element tuple (list * list) as a simple way to keep the function inputs and outputs clear

    [<TestCaseSource("TestData")>]
    member x.``Resulting list has the correct length`` (testData: float list * float list) = 
        let data, expectedResults = testData
        let states, results = 
            data 
            |> List.scan (fun (s, _) m -> SimpleExample.modifyStateAndProduceEffect s m) (1., 0.) 
            |> List.tail |> List.unzip 
        results |> should haveLength expectedResults.Length

    [<TestCaseSource("TestData")>]
    member x.``Results in list approximiately match expected results`` (testData: float list * float list) = 
        let data, expectedResults = testData
        let states, results = 
            data 
            |> List.scan (fun (s, _) m -> SimpleExample.modifyStateAndProduceEffect s m) (1., 0.) 
            |> List.tail |> List.unzip
        approxEqual results expectedResults 1e-9 |> should be True

Note, the F# will frequently not have enough information to infer the type of data being supplied to the unit tests. You will need specify it as above: (testData: float list * float list)

A Real-World Example

Here is a real-world example. The system under test is a functional called processTick. This function is part of a larger Order Inference algorithm designed to process financial data. Specifically, this function determines what changes in buy and sell sizes took placed to generate the tick. It is one step in reversing engineering the flow of order book information.

1: 

    let processTick orderBook tick =

It has the following signature

    val processTick : orderBook:OrderBook -> tick:Tick -> OrderBook * Order list

It takes an OrderBook and Tick and returns a tuple consisting of an OrderBook and a list of type Order. This is the same pattern as above. We have a state: OrderBook. That state is modified by an event: Tick. And that generates some information in the form of a list of type Order

A Tick usually consists of one of two types; Quote or Trade. Here, we use a record to model each of them. A Quote can either be on the Bid side or Ask side. So a Tick is either a Bid quote, an Ask quote, or a Trade (trades have no side). The overall strucuture is succinctly expressed as a discriminate union.

module OrderInference = 

    type Trade = { time:DateTime; price:float; size:int; }

    type Tick = 
        | Bid   of Quote
        | Ask   of Quote
        | Trade of Trade

Unit testing multiple permutations of a sequences of events

Below I have include three types of test cases. The first tow a fialy straight forward. The third case is where I find combining TestCaseSource and list comparisions so effective.

In case 3, we feed in three event; A Trade, then Bid update and an Ask update. The order here matters. The algorithm needs to be designed to handle the various permutation. Lists used inside TestCaseSource allow us to easily create and test various sequence to see how our algorithm will perform.

Furthermore, using TDD we can begin with simple cases, as in case 1, and then build towards the exteme edge case. Because the unit test rely on list comparision, any arbitray length of events can be tested uses the same set of unit tests.
